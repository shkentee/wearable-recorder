# コード監査 引継ぎ — wearable-recorder

- **日付**: 2026-06-27 (JST)
- **作成**: AI生成（Claude / コードレビュー）
- **対象**: ファームウェア `app/`（omiへのパッチ重畳, C 約2,000行）+ `firmware/`（scratch, C）/ モバイル `app_mobile/`（Flutter 約6,000行）
- **観点**: コード破綻・stuck（固着）・デッドロック・レース・リソースリーク
- **総括**: ハードな無限デッドロックは無い。ただし「片側が応答を待ち続けて固まる（stuck）」系が複数。**最大リスクは SDファイル転送（firmware ⇄ app）の両端に排他制御が無いこと**。

---

## 対応順（推奨）

1. **firmware ①** ストレージ転送の状態レース（土台。app修正より先に入れる）
2. **app ②** SD同期ループの例外保護
3. **app ③** 切断時クリーンアップ（`_stateSub` 未購読）
4. 以降は 🟠→🟡→🟢 の順で

この①②を直すと「同期が無言で止まる」主要経路が塞がる。

## 🔴 最優先（stuck・同期停止）

### ① ストレージ転送の状態レース（firmware + app の両端で同根）
- **firmware**: `app/src/wr_storage_service.c:84,236,245`
  - `wr_state`（plain enum）を BLE RXスレッド（`ctrl_write`）とシステムワークキュー（`list_fn`/`fetch_fn`）から**排他なしで読み書き**。
  - LIST 配信中に FETCH が来ると、走行中の `list_fn` がループを抜けて **236行で無条件に `WR_ST_IDLE` へ戻す** → 直後に走る `fetch_fn` が `wr_state != FETCHING` で**即return → FETCH が黙って捨てられる**。
  - スマホ側はデータを待ち続け、ウォッチドッグ時間まで固着。
- **app**: `app_mobile/lib/services/wr_storage_client.dart`（単一 `_session`/`_sub`）と `wr_sd_sync.dart` の `_drainOnce`/`_drainClosedFile` が同セッションを共有。重複ドレインで `_sub` を相互キャンセルし、どちらも正常完了しない。
- **対策**:
  - firmware: `wr_state != IDLE` のとき新コマンドはエラーで弾く。完了ハンドラは「自分が握った状態」だけをリセット（どのopか判定してから戻す）。
  - app: ストレージセッションに mutex を導入し、全コマンドを直列化。

### ② SD同期ループが例外1発で永久停止（app）
- `app_mobile/lib/services/wr_sd_sync.dart:368-466 / 473-545`
  - `start()` が `_loop()`/`_uploadLoop()` を `await`もcatchも無しで投げっぱなし。
  - `SharedPreferences.getInstance()` / `Connectivity()` / outboxディレクトリ作成 等が投げると、**未捕捉非同期例外でループが死亡 → 接続中ずっと同期が無言で停止**（再起動するまで復帰しない）。
- **対策**: 各ループ本体を try/catch で囲み、例外時は再スケジュール（loopが1回の例外で終了しないように）。

### ③ 切断時のクリーンアップ欠落で状態が固着（app）
- `app_mobile/lib/services/wr_ble_device.dart:29,78`
  - `_stateSub` を宣言し disconnect で cancel しているが、**一度も `state` を listen していない（dead field）**。
  - 予期せぬリンク切断で `_liveMonitor`/`_sink`/ストレージ `_session` が後始末されず、`isConnecting` 系フラグが true 固着 → 以後の再接続をブロック。さらに死んだ characteristic に `setNotifyValue(false)` を呼ぶ。
- **対策**: `_stateSub = _device.state.listen(...)` を実装し、`disconnected` で teardown 実行。

## 🟢 低・備考

- **`app/src/wr_chunk.c:129`** — epoch名衝突時に `sync_time_secs++` を**恒久的に進めて時刻基準をずらしている**。衝突プローブはローカル offset で行うべき。
- **`app/src/wr_chunk.c:116-131`** — 衝突プローブが mutex 取得前（現状は単一workqueueで直列なので安全だが、将来の書き込み追加で脆弱）。
- **`firmware/` ディレクトリは別物** — `firmware/src/main.c` は LED を `while(1)` で点滅させるだけの "scratch v0" bring-up。製品ファームは `app/` ＋ omiパッチ群。**2系統が同居しており混乱の元** → dead/scratch と明示推奨（README等）。
- パッチ群に `0044/0045` の revert と SD書き込み修正が大量（`0036`〜`0047`）。**SD書き込み経路は omi 側で不安定だった履歴**あり。今回のC新規ファイルの範囲外だが、要継続監視。
- `wr_battery_service_init()` は patch `0004-battery-service-init.patch` で正しく統合済み（**dead codeではない**、調査済み）。
- `app/src/wr_boot_cmd.c` のリングバッファはオーバーフロー無し（調査済み・安全）。
